맞아. 발표용 포장이 아니라 실제로 이 문제를 어떻게 풀어야 하는가로 보면, 먼저 구조를 분리해야 해요.
지금 헷갈리는 이유는 “IK, 경로계획, 최적화, PD 제어, 동역학, 충돌 회피, 숟가락 자세 유지”가 한 덩어리로 섞여 있기 때문입니다.
로봇 제어 시스템은 보통 이렇게 층을 나눕니다.

1. **Modeling**
   로봇, 숟가락, 식판, 음식, 사용자 입 위치, 제약 조건 정의
2. **Task Planning**
   어떤 phase를 거칠지 결정 (예: approach -> scoop -> lift -> carry -> mouth -> retreat)
3. **Trajectory Optimization / Motion Planning**
   각 phase에서 어떤 경로 $q(t)$, $x(t)$를 따라갈지 계산
4. **Tracking Control**
   계산된 $q_{ref}(t)$, $dq_{ref}(t)$를 실제 로봇이 따라가게 함 (예: PD, PID, computed torque, impedance control)
5. **Safety / Monitoring**
   실행 중 충돌, 관절한계, 특이점, 숟가락 기울기 검사 및 보정

여기서 중요한 점은:
> 최적화는 PD 제어를 대체하는 것이 아니라, PD 제어가 따라갈 “좋은 reference trajectory”를 만들어주는 단계다. 즉 최적화와 제어기는 경쟁 관계가 아닙니다.

전체 문제를 다시 정의하면 우리 과제는 결국 이것입니다.
**숟가락 끝이 음식 위치에 접근해서, 식판/로봇/관절 제약을 어기지 않고, 적절한 자세로 음식을 뜨고, 흘리지 않도록 들어올린 뒤, 사용자의 입 근처까지 안전하게 이동하는 trajectory를 생성하고 추종하는 문제.**

그러면 핵심 변수는 세 가지입니다.

* $q(t)$: 로봇 관절각.
* $x_{\text{spoon}}(t) = FK(q(t))$: 숟가락 tip, head, bowl, normal, forward 방향.
* $u(t)$: 실제 제어 입력. 시뮬레이션에서는 joint target, 실제 로봇에서는 torque/velocity/position command일 수 있음.

## 1. Modeling은 어떻게 해야 하나

먼저 모델링은 세 층으로 나눠야 합니다.

### 로봇 모델

$$q \in \mathbb{R}^n$$
$$x = FK(q)$$
$$J(q)=\frac{\partial x}{\partial q}$$
필요한 것은: joint limit, link geometry, spoon tip 위치, spoon head 위치, spoon normal 방향, spoon forward 방향, Jacobian, singularity metric
현재 프로젝트는 MuJoCo XML을 쓰기 때문에 로봇 모델은 이미 꽤 갖춰져 있습니다. 다만 “무엇을 제약으로 볼지”를 명확히 다시 정의해야 해요.

### 숟가락 모델

숟가락은 단순 end-effector가 아닙니다. 식사 보조에서는 tip만 맞추면 안 되고, head와 자세가 중요합니다.
예를 들어:
$$p_{\text{tip}}(q), \quad p_{\text{head}}(q), \quad n_{\text{spoon}}(q), \quad f_{\text{spoon}}(q)$$
여기서 $n_{\text{spoon}}$은 수평 유지에 쓰고, $f_{\text{spoon}}$은 숟가락 진행 방향/yaw에 씁니다.
head-down은 이렇게 정의할 수 있습니다.
$$h(q) = z_{\text{wrist}}(q) - z_{\text{head}}(q)$$
$$h(q) > 0$$
이면 숟가락 head가 wrist보다 아래에 있다는 뜻입니다.

### 환경 모델

식판, 음식, 입 위치는 task constraint입니다.

* 식판 바닥 높이
* 식판 경계
* 음식 위치
* 사용자의 입 위치
* 충돌 금지 영역
* 허용 가능한 접근 방향

이걸 전부 “if문”으로만 두면 시스템이 부서지기 쉽습니다. 가능하면 cost 또는 constraint로 표현해야 합니다.

## 2. 제어기는 보통 어떻게 설계하나

일반적인 구조는 이렇습니다.

```text
trajectory generator   -> q_ref(t), dq_ref(t), ddq_ref(t)
tracking controller    -> motor command
robot/simulation       -> q(t), dq(t)
feedback               -> error correction
```

가장 기본은 PD 제어입니다.
$$u = K_p(q_{\text{ref}}-q) + K_d(\dot q_{\text{ref}}-\dot q)$$
만약 torque control이면:
$$\tau = K_p(q_{\text{ref}}-q) + K_d(\dot q_{\text{ref}}-\dot q)$$
더 고급이면 computed torque control:
$$\tau = M(q)\ddot q_{\text{ref}} + C(q,\dot q)\dot q + g(q) + K_p e + K_d \dot e$$

하지만 지금 프로젝트에서는 처음부터 torque-level 동역학 제어로 갈 필요는 없습니다.
현재는 MuJoCo에서 position actuator 또는 joint target replay에 가까우므로, 현실적인 구조는:

1. trajectory optimization으로 $q_{ref}(t)$ 생성
2. position/PD controller로 $q_{ref}(t)$ 추종
3. 실행 중 safety monitor로 위험 상태 감지
입니다. 즉 PD는 계속 씁니다. 다만 PD가 따라갈 trajectory를 더 잘 만들어야 합니다.

---

## 3. 최적화는 어디에 들어가나

최적화는 크게 세 위치에 들어갈 수 있습니다.

### A. IK 최적화

한 순간의 목표 pose에 대해 $q$를 찾음.
$$\min_q \|p(q)-p^*\|^2 + w_R\|R(q)-R^*\|^2 + w_l\phi_{\text{limit}}(q) + w_s\phi_{\text{sing}}(q)$$
이건 “한 점”을 푸는 문제입니다. 현재 코드의 DLS IK가 이 범주입니다.

* **장점**: 구현 쉬움, 빠름
* **단점**: keyframe 사이 경로를 보장하지 않음, 전체 동작의 smoothness를 직접 보지 않음

### B. Trajectory optimization

전체 구간의 $q_{0:N}$을 한 번에 찾음.
$$\min_{q_{0:N}} J(q_{0:N})$$
이게 우리 문제에 가장 중요합니다. 왜냐하면 식사 보조는 한 pose가 아니라 연속 동작이기 때문입니다.

### C. Online MPC / safety optimization

실행 중 짧은 horizon을 계속 다시 최적화.
$$\min_{q_{k:k+H}} \sum_{i=k}^{k+H}\ell(q_i,q_i^{ref})$$
이건 최종적으로 가면 좋지만, 처음부터 하기에는 부담이 큽니다.

따라서 추천 순서는:

1. IK 최적화 정리
2. 특정 phase에 trajectory optimization 적용
3. 전체 phase로 확장
4. 필요하면 MPC 추가

## 4. 이 과제에서 가장 먼저 최적화해야 할 것

제 생각에는 IK 자체보다 trajectory를 먼저 봐야 합니다.
현재 방식은 대략:
`pre pose IK` -> `engage pose IK` -> `drag start IK` -> `drag end IK` -> `lift IK`
그 사이를 joint-space interpolation.

이 구조의 문제는: **각 점은 괜찮아도, 점 사이 경로가 나쁠 수 있다.**
예를 들어:

* 중간에 spoon tilt가 커짐
* 식판과 충돌함
* 관절한계 근처를 지나감
* 특이점에 가까워짐
* 움직임이 급격함
* head-down이 scoop 중 유지되지 않음

그래서 가장 자연스러운 해결은:
**keyframe 사이를 단순 보간하지 말고, 그 사이의 trajectory 자체를 최적화한다.**
즉 $q_0, q_1, ..., q_N$을 decision variable로 둡니다.

## 5. Cost는 어떻게 설정해야 하나

cost는 “무엇이 좋은 동작인가”를 수학적으로 쓴 것입니다.
우리 문제의 cost는 네 묶음이면 충분합니다.
$$J = J_{\text{task}} + J_{\text{spoon}} + J_{\text{safety}} + J_{\text{smooth}}$$

### Task cost

숟가락이 원하는 위치를 따라가게 하는 항.
$$J_{\text{task}} = \sum_{k=0}^{N} w_p(k)\|p_{\text{tip}}(q_k)-p_k^{ref}\|^2$$
scoop 중에는 tip이 식판 표면 근처를 따라가야 하고, mouth delivery에서는 mouth target으로 가야 합니다.

### Spoon posture cost

숟가락 자세를 조절하는 항.
$$J_{\text{level}} = \sum_k w_n(k)\|n_{\text{spoon}}(q_k)\times n_k^{ref}\|^2 \quad \text{(수평 유지)}$$
$$J_{\text{forward}} = \sum_k w_f(k)\|f_{\text{spoon}}(q_k)\times f_k^{ref}\|^2 \quad \text{(진행 방향 유지)}$$
$$J_{\text{head}} = \sum_k w_h(k)\max(0,h_k^{ref}-h(q_k))^2 \quad \text{(head-down 유지)}$$

여기서 phase별로 weight가 달라져야 합니다.

* **pre/engage**: head-down 중요
* **drag**: head-down + 진행 방향 중요
* **lift/carry**: level 중요
* **mouth**: level + 접근 방향 중요

### Safety cost

관절 한계, 특이점, 충돌 회피.
$$J_{\text{limit}} = \sum_{k,i} \left[ \max(0,q_{k,i}-(q_i^{max}-m_i))^2 + \max(0,(q_i^{min}+m_i)-q_{k,i})^2 \right]$$
$$J_{\text{sing}} = \sum_k \max(0,\epsilon-\sigma_{\min}(J(q_k)))^2$$
충돌은 거리 함수가 있으면:
$$J_{\text{col}} = \sum_k \max(0,d_{\min}-d(q_k))^2$$
거리 함수가 어렵다면 초기에는 MuJoCo contact 여부를 hard rejection으로 써도 됩니다.

### Smoothness cost

부드러운 움직임.
$$J_{\text{vel}} = \sum_{k=1}^{N} w_v\|q_k-q_{k-1}\|^2$$
$$J_{\text{acc}} = \sum_{k=1}^{N-1} w_a\|q_{k+1}-2q_k+q_{k-1}\|^2$$
이 항이 있어야 PD가 추종하기 쉬운 trajectory가 나옵니다.

## 6. Hard constraint와 soft cost를 나눠야 함

모든 걸 cost로만 넣으면 위험합니다.
예를 들어 “충돌하면 cost가 커짐”은 최적화가 실패하면 충돌할 수 있다는 뜻입니다.
그래서 다음처럼 나누는 게 좋습니다.

**Hard constraint:**

* joint limit
* self/environment collision
* 식판 바닥 관통
* mouth safety distance
* 너무 작은 singular value
* 숟가락 수평 오차

**Soft cost:**

* 목표 위치 오차
* head-down 정도
* 움직임 부드러움
* 선호 posture

현실적으로 처음 구현에서는 hard constraint를 완벽한 nonlinear constraint로 넣기 어렵습니다.
그래서 1차 구현은:
**soft penalty로 최적화 -> 결과 trajectory를 validation -> 실패하면 penalty weight 증가 또는 후보 reject**
이렇게 해도 됩니다.

## 7. 추천하는 실제 시스템 구조

내가 이 프로젝트를 정리한다면 이렇게 설계합니다.

* **[Task Planner]**: food position, mouth position을 보고 phase target 생성
* **[Initial Trajectory Generator]**: 현재 방식의 DLS IK + keyframe interpolation으로 초기 q trajectory 생성
* **[Trajectory Optimizer]**: 초기 trajectory를 받아서 $q_{0:N}$ 최적화 (cost = task + spoon posture + safety + smoothness)
* **[Validator]**: joint limit, collision, singularity, tilt, head_drop 확인
* **[Tracking Controller]**: PD/position control로 $q_{ref}(t)$ 추종
* **[Runtime Monitor]**: 실행 중 contact, tilt, q error, singularity 감시. 위험하면 stop 또는 replan

이 구조가 중요합니다. 최적화가 모든 걸 하는 게 아닙니다.
최적화는 “좋은 경로”를 만들고, PD는 “그 경로를 따라가게” 하고, monitor는 “위험하면 막는” 역할입니다.

---

## 8. 단계별 개발 계획

현실적으로는 이렇게 가는 게 좋습니다.

### Step 1: 현재 시스템을 분석 가능한 baseline으로 만든다

먼저 각 trajectory마다 다음 metric을 저장해야 합니다.
`success/fail`, `max position error`, `max tilt`, `min head_drop`, `min joint margin`, `min \sigma_{\min}`, `max condition number`, `contact count`, `trajectory length`, `acceleration cost`
이게 없으면 최적화가 좋아졌는지 판단할 수 없습니다.

### Step 2: 한 구간만 trajectory optimization 한다

처음부터 전체 sequence 하지 말고 하나만 고릅니다.
추천: `drag_start -> drag_end` (이유는 scoop 품질과 관련이 크고, head-down/식판 충돌/방향 유지가 모두 들어가기 때문입니다.)

### Step 3: decision variable을 $q_{0:N}$으로 둔다

초기값은 기존 smoothstep 보간.
$$q_k^{init} = \text{interp}(q_{\text{start}}, q_{\text{end}})$$
최적화는 이것을 조금씩 수정합니다.

### Step 4: cost를 최소 구성으로 시작한다

처음부터 다 넣지 말고:
$$J = w_pJ_{\text{task}} + w_hJ_{\text{head}} + w_lJ_{\text{limit}} + w_sJ_{\text{sing}} + w_aJ_{\text{smooth}}$$
이 정도로 시작합니다.

### Step 5: validation으로 비교한다

최적화 전후를 같은 조건에서 비교합니다.

```text
baseline interpolation:
  max_tilt = ...
  min_sigma = ...
  min_head_drop = ...
  contact = ...

optimized trajectory:
  max_tilt = ...
  min_sigma = ...
  min_head_drop = ...
  contact = ...
```

이게 진짜 문제 해결입니다.

## 9. 나중에 동역학까지 넣는다면

지금은 kinematic trajectory optimization이 먼저입니다.
하지만 나중에 동역학을 넣으면 이렇게 확장됩니다.

* 상태: $x_k = [q_k,\dot q_k]$
* 입력: $u_k = \tau_k$
* 동역학: $x_{k+1}=f(x_k,u_k)$
* cost: $J = \sum_k \ell(x_k,u_k)$

여기에 torque penalty도 추가합니다.
$$J_u = \sum_k w_u\|u_k\|^2$$
하지만 이건 훨씬 어렵습니다. 현재 프로젝트에서는 position-controlled robot처럼 다루고, $q_{0:N}$ 최적화를 먼저 하는 게 맞습니다.

---

## 정리하면

이 과제를 제대로 풀려면 이렇게 생각하면 됩니다.

1. **모델링**은 로봇/숟가락/환경/제약을 수학적으로 정의하는 단계다.
2. **최적화**는 그 모델 위에서 좋은 reference trajectory를 찾는 단계다.
3. **PD 제어**는 그 reference trajectory를 실제 로봇이 따라가게 하는 tracking 단계다.
4. **safety monitor**는 최적화와 제어가 놓친 위험을 실행 중 감시하는 단계다.

**따라서 당장 가야 할 방향은:**
현재 DLS IK + 보간 구조를 baseline으로 둔다.
$\rightarrow$ 특정 phase의 q trajectory 전체를 최적화한다.
$\rightarrow$ cost는 task, spoon posture, safety, smoothness로 나눈다.
$\rightarrow$ 최적화된 $q_{ref}(t)$를 기존 PD/position controller로 추종한다.
$\rightarrow$ metric으로 실제 개선을 확인한다.

이게 가장 현실적이고, 프로젝트를 진짜로 앞으로 밀 수 있는 길이라고 봅니다.
